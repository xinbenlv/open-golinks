# 链接编辑权限控制和登录页面

**Date**: 2026-02-10
**Priority**: P0（安全漏洞修复）
**Status**: 📋 Planning

## Context（背景）

当前系统存在安全漏洞：匿名创建的链接（`ownerId = null`）可以被任何未登录用户修改和删除。

**用户需求**：
1. ✅ 第一次创建链接可以未登录（已支持）
2. ❌ 创建后要修改/删除必须登录（当前有漏洞）
3. ❌ 删除后重新创建同样的 slug 也需要登录（当前未检查）

**用户选择的方案**：
- 使用 Supabase Auth UI 组件库（支持 Magic Link、Google OAuth、邮箱密码）
- 已删除的 slug 只有原所有者可以重用
- 匿名链接必须通过 claim API 声明所有权后才能编辑

## User Journey（用户旅程）

### Journey 1: 匿名创建后尝试编辑

**当前行为**（❌ 有漏洞）：
1. 用户未登录访问 `/create`
2. 创建链接 `go/test` → https://example.com
3. 访问 `/edit/test`
4. 可以直接修改链接（无需登录）⚠️ 任何人都可以修改

**期望行为**（✅ 安全）：
1. 用户未登录访问 `/create`
2. 创建链接 `go/test` → https://example.com
3. 访问 `/edit/test`
4. **自动重定向到** `/auth/login?returnTo=/edit/test`
5. 用户登录（Magic Link / Google / 邮箱密码）
6. **登录成功后返回** `/edit/test`
7. **看到提示**："这是一个匿名链接，要修改请先声明所有权"
8. **点击"声明所有权"按钮**
9. **声明成功后**可以编辑链接

### Journey 2: 尝试修改他人的链接

**当前行为**（⚠️ 检查不完整）：
1. 用户A登录并创建链接 `go/test`
2. 用户B登录并访问 `/edit/test`
3. 可以看到编辑表单（但提交时会失败）

**期望行为**（✅ 清晰的错误）：
1. 用户A登录并创建链接 `go/test`
2. 用户B登录并访问 `/edit/test`
3. **页面显示**："你没有权限编辑这个链接"（无法提交表单）

### Journey 3: 删除后重新创建同一个 slug

**当前行为**（❌ 无检查）：
1. 用户A创建并删除链接 `go/test`
2. 用户B（或未登录用户）尝试创建 `go/test`
3. 可以成功创建（占用了别人的 slug）

**期望行为**（✅ 严格检查）：
1. 用户A创建并删除链接 `go/test`
2. 用户B尝试创建 `go/test`
3. **API 返回 403**："此 slug 之前被使用并删除，只有原所有者可以重用"
4. 用户A尝试创建 `go/test`
5. **API 返回 200**："链接重新激活成功"（而非创建新链接）

### Journey 4: 匿名用户尝试删除匿名链接

**当前行为**（❌ 有漏洞）：
1. 用户未登录创建链接 `go/test`
2. 任何未登录用户都可以删除它

**期望行为**（✅ 必须声明所有权）：
1. 用户未登录创建链接 `go/test`
2. 尝试删除时 **API 返回 403**："匿名链接必须先声明所有权才能删除"
3. 用户登录并声明所有权
4. 现在可以删除链接

## 登录页面功能

### 支持的认证方式
- ✅ **Magic Link**（无密码邮箱登录）
  - 用户输入邮箱 → 收到登录链接 → 点击即可登录
- ✅ **Google OAuth**
  - 一键使用 Google 账号登录
- ✅ **邮箱/密码**
  - 传统登录方式（备选）

### 登录流程
1. 访问 `/auth/login`（可带 `returnTo` 参数）
2. 选择登录方式
3. 完成认证
4. **自动重定向到** `returnTo` 参数指定的页面（默认 `/dashboard`）

## Success Criteria（成功标准）

✅ **安全性**：
1. 匿名链接无法被未声明所有权的人修改/删除
2. 已删除的 slug 只能由原所有者重用
3. 所有权限检查在 API 层执行（UI 重定向只是辅助）

✅ **用户体验**：
1. 登录页面支持 3 种认证方式
2. 登录后自动返回原页面
3. 权限错误消息清晰友好
4. Claim 提示显眼且易操作

✅ **功能完整性**：
1. 所有 API 端点返回正确的 HTTP 状态码和错误代码
2. 审计日志记录所有操作
3. 所有测试通过
